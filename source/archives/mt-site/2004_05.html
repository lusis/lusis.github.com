Why I'll never buy another Linksys Product

I won't ever buy another linksys product.

It's pretty simple really. I'm tired of dealing with the Linksys idea of changing something as essential as the chipset without changing the model number.

This came up at work as we are working to migrate our workstations to Linux.

Our current install base of well over 600 Windows PCs with Linksys WMP11 cards is useless under Linux. You see Linksys has decided in thier wisdom to continually change the chipset used in these cards and not let anyone know (not even on the box) that the cards aren't the same.

The older models of WMP11 (version 1 and 2) work fine under Linux. The newer cards (version 3) do not.

At this point we told our Linksys rep we were dropping them in favor of D-Link. The USB wireless keyfobs work under Linux, are cheaper to replace (no hardware tech needed on site) and D-Link has assured me that since this is the only product line they have that works under Mac OSX, they plan on keeping it. I made my distaste with Linksys VERY clear to our new D-Link rep.

But the Linksys problem isn't simply applicable to the wireless card product line. It also leaks over into the WAP line as well. Currently at each of these locations we have a Linksys WAP11 access point. The model we have seems to have a nasty bug whereby you MUST input the Netmask and the gateway in the IP configuration screen in reverse order (i.e. the Netmask goes in the gateway section and vice-versa)!

This is insane. So I check the Linksys website for updated firmware that would surely resolve this minor but seemlingly easy flaw to fix. Check this page:

http://www.linksys.com/download/

They have 4 different versions of this damn access point with the same name!

How am I supposed to know which of the 4 I have when I have over 300 of these guys out there? It seems the only way is to compare the firmware version on the AP with the firmware on each model. IF the numbers are close enough you can bet that you found the right one.

So I go through this and find out that Linksys hasn't updated the firmware for the AP we have since August of last year! I'm guessing they don't want to bother.

Meanwhile I'm taking my business from Linksys to D-Link (and Netgear if need be) to avoid this kind of shoddy business. I expect more from Cisco. Then again I'm slowly replacing all my cisco switches with HP5308XL and 2924 models.

Commercial software ebuild for Gentoo

Problem: I need to run some commercial software on Gentoo that doesn't have an ebuild and comes in RPM format only.

Solution: This "Call to Arms" on the Gentoo Chat forums

The first ebuild is a modified version of an ebuild that sandymac[at]gentoo.org did for the Tivoli Storage Manager Backup Client. His version was for 5.1.5. We have 5.2.2 running at our datacenter right now. The server runs on RHAS 3.0 and all of the clients, save three, are RedHat boxes.

The Gentoo clients are:

Firebird Database Server: for our Data Warehouse
Cups Server A and Cups Server B: Handles all of the printing for our web application. (Our locations are not allowed to print things themselves only server initiated printing is allowed)

These three servers need client access to the TSM server for backups. My original thought was to use rsync to one of the redhat boxes and store the files there for backup but that wasn't elegent enough. It also adds one more step that could break along the way. Considering I don't want to get a call from a junior admin at 4AM because he can't figure out how to restore the DW database or our printer pool, I'll use TSM and create nodes for the Gentoo servers.

I'm creating the entry here so I can provide a full tarball of a portage overlay. I'm also looking for more widespread commenting on the whole process.
